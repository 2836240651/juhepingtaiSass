# Temu 一键刷新爬取 — 需求分析

## 背景

原架构为运维手动执行 `py crawl.py --tenant-id N` 写入 SQLite，前端仅通过 Java GET 接口读取缓存数据。用户在 Temu 运营页无法主动触发数据同步。

## 目标

- 用户在 Temu 运营页点击「刷新数据」→ Java 异步调用 Python 爬虫 → 入库后前端自动重载。
- 交互模式：**异步任务 + 前端轮询**（Playwright 通常 1~3 分钟）。
- 权限：**Boss + 有 Temu 平台/店铺权限的员工**。
- 租户隔离：JWT `tid` → Python `--tenant-id`。

## 功能需求

| ID | 需求 |
|----|------|
| FR-01 | `POST /api/temu/crawl` 触发爬取，返回 `job_id` |
| FR-02 | `GET /api/temu/crawl/{jobId}` 查询任务状态 |
| FR-03 | 同租户同时仅 1 个 running 任务，重复触发 409 |
| FR-04 | 员工无 Temu 权限时 403 |
| FR-05 | 前端轮询 2s，最长 5min，成功 Toast 并重载数据 |
| FR-06 | Python `--json` 结构化输出供 Java 解析 |

## 非功能需求

- 单任务超时 300s（可配置）
- 生产环境 `allow-seed: false`
- 服务重启时将 pending/running 任务标为 failed

## 非目标

- 定时调度、多店铺并行爬、Express 参与
