# 通用竞店监控框架需求清单

## 1. 背景

当前项目已经具备以下基础能力：

- Java 后端可提供 Temu 相关业务 API。
- Python 侧已有 Temu 抓取脚本与数据处理逻辑。
- Java 与 Python 共用同一套 SQLite 数据库。
- 前端已有竞店分析页面与本地演示模式。

现阶段的核心问题：

- 竞店分析偏一次性执行，缺少长期监控任务模型。
- 缺少“是否已有最新数据”的统一查询口径。
- 缺少手动触发、定时调度、任务历史、快照历史的一体化能力。
- Temu 逻辑与平台逻辑耦合，不利于扩展 Amazon、TikTok Shop、Walmart、Shopify。

因此需要建设一套“通用竞店监控框架”，先支撑 Temu，后续复用到多平台。

## 2. 建设目标

### 2.1 业务目标

- 支持按店铺维度建立长期监控目标。
- 支持手动触发与自动调度两种监控模式。
- 支持前端与 Java 后端查询店铺是否已有最新快照。
- 支持识别“近期上新”和“异常销售”两类核心信号。
- 支持输出 Excel 与 Markdown 报表。
- 支持保留历史快照、任务记录与报表路径。

### 2.2 技术目标

- 复用现有 Java + Python + SQLite 架构。
- 用通用 monitor 表结构替代平台专用临时表。
- 平台抓取逻辑通过 adapter 插件化接入。
- 后续新增平台时，尽量只新增 Python adapter 与少量映射配置。

## 3. 角色与场景

### 3.1 角色

- 运营/Boss：配置监控目标、查看结果、手动触发。
- Java 后端：查询最新状态、对外提供统一接口。
- Python Worker：轮询任务、执行抓取、分析、落库、生成报表。

### 3.2 关键场景

1. 查询最新数据
   - 调用方查询某个店铺是否已有新鲜数据。
   - 若命中 freshness 规则，直接返回最新结果。
   - 若未命中，返回 miss 状态和是否允许手动触发。

2. 手动触发监控
   - 用户手动对某个目标发起抓取。
   - 系统创建 `pending` job。
   - Worker 消费任务，生成快照、信号与报表。

3. 自动调度监控
   - 用户为目标配置 interval 或 cron。
   - Java Scheduler 周期扫描计划并创建 job。
   - Worker 后续异步处理。

4. 历史查看
   - 用户查看目标快照历史、任务历史、报表产物路径。

## 4. 范围定义

### 4.1 本期纳入范围

- 通用 monitor 目标、计划、任务、快照、商品快照、信号表。
- Java Monitor API。
- Java Scheduler。
- Python Monitor Worker。
- Temu 平台 adapter。
- Excel/Markdown 报表落地与路径记录。
- 前端竞店页面对 monitor API 的联调。

### 4.2 后续扩展范围

- Amazon / TikTok Shop / Walmart / Shopify adapter。
- 监控告警。
- 批量任务编排。
- 趋势分析与同比/环比。

### 4.3 暂不纳入

- 分布式抓取集群。
- Kafka / RabbitMQ 等消息队列。
- 多机房调度。
- 复杂审批流。

## 5. 功能需求

### 5.1 监控目标管理

- 支持新增目标：
  - `platform`
  - `target_type`
  - `label`
  - `target_url`
  - `freshness_minutes`
  - `crawl_strategy`
- 支持编辑目标。
- 支持删除目标。
- 支持按平台/状态列出目标。

### 5.2 监控计划管理

- 支持配置 `interval` 和 `cron` 两种计划。
- 支持启停计划。
- 支持设置单次抓取最大商品数。
- 支持设置失败重试上限。
- 支持返回 `next_run_at` / `last_run_at`。

### 5.3 手动触发

- 支持 `POST /trigger` 创建任务。
- 返回 `job_id` 和当前状态。
- 同一目标存在 `pending/running` 任务时返回冲突。

### 5.4 最新状态查询

- 支持返回：
  - `has_fresh_data`
  - `latest_snapshot_id`
  - `latest_snapshot_at`
  - `latest_job_status`
  - `can_trigger_now`
  - `reason`
  - `summary`
  - `signals`
  - `artifacts`

### 5.5 历史查询

- 支持返回最近快照历史。
- 支持返回最近任务历史。
- 支持前端展示调度与执行情况。

### 5.6 抓取结果与分析

- 每次成功执行必须生成一个快照头记录。
- 每个快照下保存商品明细。
- 至少识别两类信号：
  - `recent_launch`
  - `sales_outlier`
- 保存 Excel/Markdown 报表路径。

## 6. 非功能需求

- 租户隔离：所有 monitor 数据按 `tenant_id` 隔离。
- 一致性：快照、商品、信号、任务状态需要同链路落库。
- 可观测性：任务需具备明确状态、错误码、错误消息。
- 可扩展性：平台抓取逻辑解耦。
- 兼容性：首期兼容 SQLite，后续可迁移其他关系型数据库。

## 7. 验收标准

- 能创建 Temu 店铺监控目标。
- 能配置 schedule 并自动生成 job。
- `trigger` 能返回 `job_id`。
- `latest` 能正确区分 fresh 与 miss。
- Worker 能消费 `pending` 任务并回填最新快照。
- 能写入 `monitor_snapshot` / `monitor_product_snapshot` / `monitor_signal`。
- 能生成 Excel 与 Markdown 报表。
- 历史查询能返回 snapshots 与 jobs。
- 前端页面能在 backend 模式下完成目标管理、触发、状态展示、历史展示。
