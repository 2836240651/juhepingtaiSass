## 3. 对齐结果

### 3.1 已对齐的接口

- `GET /api/monitor/targets?platform=temu`
  - 前端用于加载竞店列表。
  - 已消费字段：`id`、`label`、`target_url`、`host`、`status`、`freshness_minutes`、`latest_snapshot_id`、`latest_snapshot_at`。

- `POST /api/monitor/targets`
- `PUT /api/monitor/targets/{id}`
- `DELETE /api/monitor/targets/{id}`
  - 前端已用于竞店新增、编辑、删除。

- `GET /api/monitor/targets/{id}/latest`
  - 前端已消费字段：
    - `has_fresh_data`
    - `latest_snapshot_at`
    - `latest_job_status`
    - `can_trigger_now`
    - `reason`
    - `summary.product_count`
    - `summary.new_launch_count`
    - `summary.sales_outlier_count`
    - `signals.recent_launch`
    - `signals.sales_outlier`
    - `artifacts.report_md_path`
    - `artifacts.report_xlsx_path`

- `POST /api/monitor/targets/{id}/trigger`
  - 前端已用于手动触发抓取。
  - 已处理 `202 accepted` 正常链路。
  - 已处理 `409 MONITOR_JOB_IN_PROGRESS` 冲突链路。

- `GET /api/monitor/jobs/{jobId}`
  - 前端已用于轮询任务状态。

- `GET /api/monitor/targets/{id}/history`
  - 前端已消费字段：
    - `snapshots[].snapshot_id`
    - `snapshots[].snapshot_at`
    - `snapshots[].product_count`
    - `snapshots[].new_launch_count`
    - `snapshots[].sales_outlier_count`
    - `jobs[].job_id`
    - `jobs[].trigger_type`
    - `jobs[].status`
    - `jobs[].queued_at`
    - `jobs[].finished_at`

### 3.2 本次补齐的前端行为

- backend 模式下，竞店页已切换到 `monitor` API，不再走旧 `/api/temu/competitors*` 报表查询链路。
- 当 `latest.has_fresh_data=true` 时，页面直接展示后端最新快照摘要。
- 当 `latest.has_fresh_data=false` 且允许触发时，前端会自动调用 `trigger` 并轮询 `job`。
- 当目标已有 `pending/running` 任务时，前端不会误判为空，而是等待现有任务完成后再刷新结果。
- 当 `trigger` 返回 `MONITOR_JOB_IN_PROGRESS` 时，前端会读取冲突返回中的 `job_id` 并继续轮询。
- 页面已展示：
  - Fresh/Miss 状态
  - 最新 job 状态
  - 报表产物路径
  - 快照历史
  - 任务历史
- backend 模式下无快照/无明细时，页面已提供更符合 monitor 语义的空态提示。
